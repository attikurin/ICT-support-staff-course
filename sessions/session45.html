1.1 ネットワークアーキテクチャの高度化

📐 三層ネットワーク構造（3-Tier Architecture）

💡 三層構造のメリット

  • 拡張性：各層を独立して増設・変更できる（例：教室増設時はアクセス層のみ追加）
  • 管理性：障害発生時の切り分けが容易、影響範囲が限定される
  • 性能：各層で最適な機器を選択でき、全体の性能を最大化
  • セキュリティ：層ごとにセキュリティポリシーを適用、多層防御が可能

🛡️ DMZ（DeMilitarized Zone：非武装地帯）の設計

⚠️ DMZ設計の注意点

  • 二重ファイアウォール：外部FWと内部FWで異なるベンダー製品を使用すると、脆弱性の共有を防げる
  • 通信ルールの最小化：DMZから内部への通信は必要最小限に制限（例：データベース接続のみ許可）
  • ログ監視：DMZ内サーバーへのアクセスログを常時監視し、異常を早期発見
  • 定期的な脆弱性診断：外部公開サーバーは攻撃対象になりやすいため、定期的なセキュリティ診断が必須

🔄 クラウド移行6R戦略

既存のオンプレミスシステムをクラウドに移行する際、6つの戦略（6R）から最適な方法を選択します。システムの特性、予算、期間、リスクを総合的に判断します。

戦略 内容 適用シーン（教育現場） メリット デメリット
1. Rehost
（リホスト）
"Lift and Shift"		 そのままクラウドへ移行
仮想マシンごと移動 		• 既存の校務支援システム
• レガシーな成績管理システム
• 短期間で移行が必要な場合 		• 移行が速い（数週間）
• リスクが低い
• 変更が最小限 		• クラウドの利点を活かせない
• コスト最適化不十分
• 技術的負債が残る
2. Replatform
（リプラットフォーム）
"Lift, Tinker, and Shift"		 最小限の最適化を行って移行
クラウドの一部機能を活用 		• データベースをRDSに変更
• ストレージをS3に変更
• 負荷分散をELBに変更 		• 一定の最適化
• リスクと効果のバランス
• 移行期間は中程度 		• 完全な最適化ではない
• 追加の設計が必要
• テストが必要
3. Repurchase
（リパーチェス）
"Drop and Shop"		 SaaSに乗り換え
既存システムを廃止 		• オンプレメールサーバー → Google Workspace
• 自社開発グループウェア → Microsoft 365
• ファイルサーバー → Google Drive 		• 最新機能が使える
• 運用負荷が最小
• セキュリティ向上 		• データ移行が複雑
• 操作方法の再教育
• カスタマイズ不可
4. Refactor
（リファクタ）
"Re-architect"		 クラウドネイティブに再設計
マイクロサービス化 		• 新しい学習管理システム開発
• AIを活用した個別最適化学習システム
• サーバーレスアーキテクチャ採用 		• クラウドの利点を最大活用
• 高い拡張性
• 最新技術の導入 		• 開発期間が長い（数ヶ月〜年単位）
• コストが高い
• 高度なスキル必要
5. Retire
（リタイア）		 廃止する

🔑 ゼロトラストの3大原則

  1. 決して信頼せず、常に検証する（Never Trust, Always Verify）
    • 内部ネットワークからのアクセスも含め、すべてのアクセスを毎回検証
    • 例：職員室からのアクセスでも、成績システムへのアクセスには毎回認証が必要
  2. 最小権限の原則（Least Privilege Access）
    • ユーザーには必要最小限の権限のみを付与
    • 例：担任は自分のクラスの成績のみ閲覧可能、他クラスはアクセス不可
  3. 侵害を想定する（Assume Breach）
    • 常にシステムが侵害されている可能性を前提に設計
    • 例：異常なアクセスパターンを検知したら即座にアクセスをブロック

  • ネットワーク帯域：VDIは画面転送のため、十分な帯域幅が必要（1ユーザーあたり2-5Mbps推奨）
  • 遅延対策：動画再生やマウスカーソルの遅延を最小化するプロトコル選択（VMware Blast, Citrix HDX等）
  • サーバーリソース：ユーザー数に応じたCPU・メモリの適切な設計（過不足のバランス）
  • ライセンス：VDI用のWindowsライセンス、アプリケーションライセンスの確認
  • 災害対策：VDI環境のバックアップ、冗長化構成の検討

⚠️ 要件定義でよくある失敗

  • 曖昧な表現：「使いやすいシステム」→ 誰にとって？どのような操作が？具体的な基準は？
  • 過剰な要件：「すべての機能が必要」→ 優先順位をつけて段階的導入を検討
  • 非機能要件の漏れ：機能のみ定義し、性能・セキュリティを後回し → トラブルの原因に
  • ユーザー不在：管理者のみで決定し、現場教員の意見を聞かない → 現場で使われないシステムに
  • 将来の拡張性未考慮：現在のニーズのみで設計 → 数年後に全面刷新が必要に

⚠️ 論理分離の実装方法

ネットワークの論理分離には、主に以下の方法があります：

  • VLAN分離：同一物理ネットワーク上でVLANを使用して論理的に分離（最も一般的）

🎯 EVM（Earned Value Management：出来高管理）

プロジェクトの進捗とコストを定量的に管理する手法です。計画と実績を比較し、遅延やコスト超過を早期に検知できます。

指標 略称 意味 計算式
計画価値 PV
(Planned Value)		 この時点までに完了予定の作業の予算 計画した作業量 × 予算
出来高 EV
(Earned Value)		 実際に完了した作業の価値 完了した作業量 × 予算
実コスト AC
(Actual Cost)		 実際に使用したコスト 実際の支出額
分析指標 略称 計算式 意味 判断基準
コスト効率指数 CPI
(Cost Performance Index)		 CPI = EV / AC コストの効率性 • CPI > 1.0：予算内
• CPI = 1.0：予算通り
• CPI < 1.0：予算超過
スケジュール効率指数 SPI
(Schedule Performance Index)		 SPI = EV / PV スケジュールの効率性 • SPI > 1.0：予定より早い
• SPI = 1.0：予定通り
• SPI < 1.0：遅延
完成時総コスト予測 EAC
(Estimate at Completion)		 EAC = BAC / CPI 現在のペースでの最終コスト予測 • EAC > BAC：予算超過の予測
• EAC = BAC：予算内の予測
• EAC < BAC：予算内余裕あり

📊 EVMの実践例

シナリオ：学校ICTシステム導入プロジェクト（予算5,000万円、期間12ヶ月）

現在：プロジェクト開始から6ヶ月経過

  • PV（計画価値）：2,500万円（6ヶ月で50%完了予定）
  • EV（出来高）：2,000万円（実際の進捗は40%）
  • AC（実コスト）：2,200万円（実際の支出）

分析：

  • CPI = EV / AC = 2,000 / 2,200 = 0.91 → コスト効率が悪い（予算の91%の価値しか生んでいない）
  • SPI = EV / PV = 2,000 / 2,500 = 0.80 → スケジュール遅延（予定の80%しか進んでいない）
  • EAC = BAC / CPI = 5,000 / 0.91 = 5,495万円 → このままだと495万円の予算超過

対策：

  • 要員追加によるスケジュール回復（ただしコストは増加）
  • スコープの見直し（優先度の低い機能を次フェーズに延期）
  • 作業プロセスの見直し（無駄な作業の削減）

💡 プロジェクト管理のベストプラクティス

  • 週次進捗会議：毎週決まった曜日に進捗確認、課題共有、意思決定
  • 課題管理表：発生した課題を一元管理、担当者・期限・ステータスを明確化
  • 変更管理：仕様変更は必ず影響分析（スケジュール・コスト・品質）を行い、承認プロセスを経る
  • リスク管理：プロジェクト開始時にリスク一覧を作成、発生確率と影響度で優先順位付け
  • コミュニケーション計画：誰に・何を・いつ・どのように報告するかを事前に定義
  • ドキュメント管理：すべての成果物をバージョン管理、関係者がアクセスできる共有フォルダに保存

🧠 プログラミング的思考とは

プログラミング的思考とは、「自分が意図する一連の活動を実現するために、どのような動きの組み合わせが必要で、どう組み合わせればよいか、といった論理的に考える力」です。

プログラミング的思考の4つの要素

┌─────────────────────────────────────────────────────┐
│         【プログラミング的思考の構成要素】            │
└─────────────────────────────────────────────────────┘

1️⃣ 【分解 (Decomposition)】
   複雑な問題を小さな問題に分解する
   
   例：掃除ロボットを動かす
   └→ ①ゴミの位置を検知する
      └→ ②ゴミの方向に移動する
         └→ ③ゴミを吸い取る
            └→ ④次のゴミを探す


2️⃣ 【パターン認識 (Pattern Recognition)】
   共通するパターンや規則性を見つける
   
   例：正多角形を描く
   → すべて「前に進む→決まった角度曲がる」の繰り返し
      正三角形：120度曲がる × 3回
      正方形：  90度曲がる × 4回
      正六角形：60度曲がる × 6回


3️⃣ 【抽象化 (Abstraction)】
   本質的な要素を抽出し、不要な詳細を省略する
   
   例：信号機のプログラム
   → 時間管理だけに注目
      ・色の美しさ → 不要
      ・光の強さ → 不要
      ・形や大きさ → 不要
      ・点灯時間 → 重要 ✓


4️⃣ 【アルゴリズム (Algorithm)】
   問題解決の手順を論理的に組み立てる
   
   例：数当てゲーム（1-100の数を当てる）
   
   非効率な方法：
   1から順番に試す → 最悪100回必要
   
   効率的な方法（二分探索）：
   ① 50と聞く
   ② 「もっと大きい」→ 75と聞く
   ③ 「もっと小さい」→ 62と聞く...
   → 最大7回で必ず当たる


【プログラミング的思考の活用場面】
• 運動会の準備（役割分担、スケジュール管理）
• 料理（レシピの手順化）
• 道案内（最短経路の選択）
• 整理整頓（分類ルールの作成）
• 勉強計画（目標設定、スケジュール作成）

著作権法第35条の改正（2020年）

GIGAスクール構想やオンライン授業の普及に対応し、授業目的での著作物利用が柔軟になりました。

項目 改正前 改正後（現在）
対象 教室での授業のみ 対面授業＋オンライン授業・オンデマンド配信も可能
複製 紙のプリント配布 デジタルファイルの配布も可能
公衆送信 原則不可 授業目的であれば可能（補償金支払いが必要）
範囲 必要最小限 必要最小限（変わらず）

✅ 利用できる例

  • 市販の問題集をコピーして学習支援システムにアップロード
  • 音楽の楽譜をデジタル化して授業で使用
  • Zoomで授業中に教科書を画面共有
  • 授業の録画を欠席者に視聴させる
❌ 利用できない例
  • 授業に直接関係ない娯楽目的での利用

  • 市販のドリルをすべてコピーして配布（ドリルの代替となる）

問題2：クラウド移行戦略

既存のオンプレミス校務支援システムをクラウドに移行する際、最小限の変更で短期間に移行を完了したい。この場合、6R戦略のうち最も適切な戦略はどれか。

  2. Rehost（そのままクラウドへ移行）
  3. Repurchase（SaaSに乗り換え）

正解：b

解説：Rehost（"Lift and Shift"）は、既存システムをそのままクラウドに移行する戦略で、変更が最小限で済み、短期間（数週間）での移行が可能です。Refactorは時間がかかり、Repurchaseは新しいシステムへの移行が必要で、Retireは廃止を意味します。

問題3：ゼロトラストセキュリティ

  1. 内部ネットワークからのアクセスは信頼し、外部からのアクセスのみ検証する
  2. ユーザーには必要最小限の権限のみを付与する
  3. 常にシステムが侵害されている可能性を前提に設計する
  4. すべてのアクセスを毎回検証し、決して信頼しない

正解：a

解説：ゼロトラストでは内部・外部を区別せず、すべてのアクセスを検証します。「内部は安全」という前提を排除するのがゼロトラストの基本原則です。
